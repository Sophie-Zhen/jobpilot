You're right that it looks messy. The short version: pipeline_jobs.json is a single append-only file written by several generations of discovery code, each with its own record shape and its own idea of where the "when was this posted" information goes. Nothing normalizes records afterward, and nothing ever evicts old ones. So the file is a geological record of every ingestion path you've ever run — not a clean snapshot of "current jobs."

你的直觉没错，它确实乱。一句话概括：pipeline_jobs.json 是一个只追加（append-only）的文件，被好几代发现代码先后写入，每一代都有自己的记录结构，也各自决定把"发布时间"放在哪个字段里。事后没有任何环节去统一这些记录，也从来不删除旧记录。所以这个文件像地质沉积层一样，记录了你跑过的每一条采集路径——而不是"当前职位"的干净快照。

Three things worth calling out:

三个要点：

  • merge_jobs dedups on job["id"] and nothing else — no field normalization; the same real job from T1 vs T2 won't collide (different id schemes).

  • It's append-only by design — "never removes, no TTL/staleness pruning." This one choice causes most of what you see.

3. Where the messiness comes from · 混乱的根源（5 个叠加成因）

The mess is not one bug. It's five distinct, compounding causes, each traceable to a concrete fact.

这不是单一 bug，而是五个相互叠加的成因，每一个都能对应到具体证据。

A. Multiple historical writers, divergent schemas 多代写入者，结构各异

Field coverage is segregated by era: company_slug on 100% of greenhouse/ashby records but 0% of linkedin records. posted_at on 84%, posted on only 16%, employment_type on 1 record, skills empty on 90%, description empty on 34%. A record's shape tells you which generation wrote it.

字段覆盖率按"代际"割裂：company_slug 在 greenhouse/ashby 记录中 100% 存在，但在 linkedin 记录中为 0%。posted_at 占 84%，posted 仅 16%，employment_type 只有 1 条，skills 有 90% 为空，description 有 34% 为空。一条记录的结构形状就能告诉你它是哪一代代码写的。

B. No normalization on merge 合并时不做归一化

merge_jobs dedups by id only — whatever shape a producer emits lands on disk forever. There's no "coerce to canonical schema" step, which is why A persists instead of healing.

merge_jobs 只按 id 去重——生产者发出什么形状，就永远以那个形状落盘。没有"统一为规范结构"的环节，这正是成因 A 无法被自愈、反而长期存在的原因。

C. Date info scattered across 4 inconsistent fields 日期信息散落在 4 个格式不一的字段

The single biggest reason it looks messy:

这是"看起来乱"的最大单一原因：

  • posted_at (84%): clean ISO-with-tz — 干净的带时区 ISO，但只出现在 ATS/API 记录上
  • posted (16%): a chaotic mix — ISO dates (~45), relative strings (~28), empty (4), "just now" (1) — 大杂烩：ISO 日期约 45 条、相对字符串约 28 条、空 4 条、"just now" 1 条
  • date_found (97%): clean ISO date — the anchor for relative strings — 干净 ISO 日期，是解析相对字符串的锚点
  • listed: 100% null — a dead field — 全为 null，一个死字段

And cli.py:_job_age_days only understands part of this: it reads absolute dates from posted_at and treats posted as relative. The ~43 legacy records with an absolute date in posted fall through → age=None → mislabeled "undateable" when they're really just old. That is the exact mechanism behind your 64 stale records.

而 cli.py:_job_age_days 只理解其中一部分：它从 posted_at 读绝对日期，却把 posted 当作相对字符串。那 ~43 条把绝对日期写在 posted 里的遗留记录因此匹配失败 → age=None → 被错误标记为"无法判定日期"，其实它们只是旧而已。这正是你那 64 条陈旧记录的成因机制。

D. Append-only, no eviction 只追加，从不淘汰

Never removes anything. April records persist 7 weeks later, indefinitely. (Notably seen_jobs.json does prune on 90 days — so the dedup ledger self-cleans, but the pipeline file does not.)

从不删除任何东西。四月的记录在 7 周后依然在，且会一直在。（值得注意的是 seen_jobs.json 确实有 90 天的清理——所以去重账本会自清，但 pipeline 文件不会。）

E. Rejected jobs never ledgered → perpetual churn 被拒职位从不入账 → 永久空转

digested.json records only sent jobs, not gate-rejected ones. So every digest run re-loads the same 64 stale records, re-evaluates, re-drops them — forever.

digested.json 只记录已发送的职位，不记录被闸门拒绝的。于是每次 digest 运行都会重新加载同样的 64 条陈旧记录、重新评估、再次丢弃——永远循环。

So "it looks messy" is accurate and has a precise anatomy: heterogeneous historical records (A) never normalized (B), carrying dates in four inconsistent fields the age logic only partly reads (C), in a file that never evicts (D), churned endlessly because rejections aren't ledgered (E). Your current T1/T2 paths are clean — the mess is unevicted legacy sediment plus the missing normalization/eviction/rejection-ledger layer around the merge.

所以"看起来乱"既准确、又有精确的解剖结构：异构的历史记录 (A) 从未被归一化 (B)，日期分散在四个格式不一、而年龄逻辑只读懂一部分的字段里 (C)，存在一个从不淘汰记录的文件中 (D)，又因为被拒记录不入账而被无休止地重算 (E)。你当前的 T1/T2 路径是干净的——乱的是没被清掉的历史沉积，加上合并环节周围缺失的"归一化 / 淘汰 / 拒绝入账"这一层。

5. What this means for the prune · 这对"清理"意味着什么

Deleting the 64 stale records clears the sediment today, but causes A–E mean it will slowly re-accumulate unless you also add: (1) a TTL/eviction on merge, and (2) a rejection ledger so gate-dropped jobs aren't re-evaluated forever.

删掉那 64 条陈旧记录能当下清掉沉积，但 A–E 这些成因意味着：除非你再补上 (1) 合并时的 TTL/淘汰机制，以及 (2) 一个"拒绝账本"让被闸门丢弃的职位不再被反复评估，否则它会慢慢重新堆积起来。
